## Context

Skill Runner 当前从单一仓库目录语义演进到“内建 + 用户”双目录模型。  
该模型需要同时满足三类约束：
- 内建能力开箱即用（镜像内自带，不依赖宿主机预置）
- 用户能力可插拔（可通过宿主机目录挂载持久化）
- 冲突行为可预测（同 ID 时用户覆盖内建）

本次设计是一次跨注册、安装、管理 API、管理 UI 与部署文档的横切升级。

## Goals / Non-Goals

**Goals:**
- 以固定目录名（`skills_builtin`、`skills`）建立稳定运行时约束。
- 确保同 `skill_id` 的覆盖规则确定且可测试（用户优先）。
- 确保安装/更新链路只操作用户目录，避免污染内建目录。
- 对外暴露 `is_builtin`，让前端不推断路径即可正确展示来源标识。

**Non-Goals:**
- 不引入新的路径环境变量或动态目录命名策略。
- 不在本次变更中处理历史 named volume 迁移脚本。
- 不改变执行期 `.<engine>/skills/<skill_id>` 的运行目录语义。

## Decisions

### Decision 1: 固定双目录命名并显式分层扫描

- 选择：内建目录固定为 `<ROOT>/skills_builtin`，用户目录固定为 `<ROOT>/skills`。
- 扫描顺序固定：`skills_builtin -> skills`。
- 决策理由：目录语义稳定、文档成本低、便于容器模板和 CI 一致化。
- 备选方案：
  - 仅使用 named volume：用户可见性差，不利于手动插拔。
  - 单目录混用：难以同时保证内建可用与用户覆盖可控。

### Decision 2: 同 ID 冲突采用用户优先

- 选择：后扫描到的用户目录条目覆盖内建条目。
- 决策理由：用户定制优先符合插件/本地部署预期，也便于热修复覆盖。
- 备选方案：内建优先会阻碍用户修复/替换同 ID skill，不符合可插拔目标。

### Decision 3: 包管理链路只写用户目录

- 选择：安装、更新、归档、invalid/staging 全部限定在 `skills/` 侧。
- 决策理由：避免误写镜像内内建目录，隔离“平台内容”和“用户内容”生命周期。
- 备选方案：允许覆盖写入内建目录会增加回滚复杂度与发布污染风险。

### Decision 4: 管理 API 提供 `is_builtin` 作为来源真值

- 选择：基于最终生效 skill 的实际来源路径判定 `is_builtin`。
- 决策理由：前端无需自行比对路径，直接依赖稳定布尔字段渲染。
- 备选方案：前端本地推断路径，容易与后端路径策略漂移。

### Decision 5: 管理 UI 仅基于 `is_builtin` 渲染内建标识

- 选择：`is_builtin=true` 显示标识；被用户同 ID 覆盖后标识消失。
- 决策理由：UI 与后端单一真值对齐，避免重复逻辑和不一致。

## Risks / Trade-offs

- [Risk] 用户误以为覆盖后仍能看到“内建”标识  
  → Mitigation: 明确说明“标识对应最终生效版本”，并在文档给出覆盖示例。

- [Risk] 遗漏某些仍引用 `skills/` 作为内建来源的旧路径  
  → Mitigation: 增加全文检索检查点，回归测试覆盖双目录冲突场景。

- [Risk] API 新字段被旧客户端忽略  
  → Mitigation: `is_builtin` 以向后兼容方式新增，不影响旧字段语义。

## Migration Plan

1. 发布包与镜像以 `skills_builtin/` 作为内建来源。
2. 部署模板将用户可写目录挂载到 `skills/`。
3. 管理 API/管理 UI 同步发布 `is_builtin` 与标识渲染。
4. 文档统一改写目录语义与覆盖规则。
5. 回归验证：仅内建、仅用户、同 ID 覆盖三类场景。

回滚策略：若出现严重回归，回滚到上一版本镜像与 API/前端构建；双目录下用户目录数据不丢失。
