# Skill Runner v0.6.0

发布日期：2026-04-17

版本类型：重大版本更新

## 发布摘要

`v0.6.0` 是相对 `v0.5.2` 的一次较大版本升级。

这一版本的重点不在于引入单一的新能力，而在于对运行时基础设施进行了一轮系统整理。我们重点改进了任务编排、结构化输出处理、交互式回复体验、认证流程治理、运行观测界面，以及相关的稳定性与测试能力。

对于用户而言，这一版本带来的主要变化是：任务执行结果更稳定，交互式任务更自然，错误和状态更容易理解，运行详情页也更适合用于排查问题。

## 主要更新

### 1. 结构化输出处理更加稳定

这一版本重新梳理了结构化结果的处理链路，重点改善了不同引擎输出格式之间的兼容与收敛逻辑。

- 改进了结构化输出的识别与归一化流程
- 减少了“结果已正确生成但仍被误判为需要修复”的情况
- 成功、等待补充和失败之间的边界更加清晰
- 最终结果来源的记录更加明确，便于定位问题

### 2. 多轮修复场景下的聊天展示更清晰

当模型第一次给出的最终回复不满足输出约定时，系统可能会在同一次执行中发起修复。此前这类场景容易在聊天窗口中产生多条最终回复，影响阅读和判断。

在 `v0.6.0` 中，这部分体验做了统一整理：

- 聊天窗口只突出展示最终有效的回复
- 被打回的旧版本会保留为折叠历史，而不是直接混在主消息流中
- 管理界面和 e2e 前端的显示逻辑进一步对齐

### 3. 交互式回复体验更灵活

对于需要用户继续补充信息的任务，前端回复方式做了改进。

- 在带有选项提示卡的场景下，用户仍可以直接点击建议选项
- 同时也可以输入额外的自由文本，而不再被限制在预设选项内
- 非开放文本场景下，输入框会自动切换为更紧凑的样式

这项调整主要提升了交互任务的可用性，尤其是在提示卡不能完整表达用户意图时。

### 4. 认证会话治理更加完整

这一版本对认证相关逻辑做了系统性整理，重点解决了认证等待、取消任务、会话残留以及多认证场景之间的相互影响问题。

- 认证会话的生命周期与作用域管理更加清晰
- 任务取消后，相关认证状态能够更及时地清理
- 不同认证场景之间的相互阻塞情况有所减少
- 界面上关于是否仍需认证、认证是否已失效的表现也更一致

### 5. 错误提示和状态语义更加明确

除了成功路径，这一版本也加强了失败与等待状态的区分能力。

系统现在更清楚地区分以下几类情况：

- 执行失败
- 需要重新认证
- 需要用户补充信息
- 输出格式不符合约定，需要系统修复
- 外部服务或配额导致的限制

这有助于用户更快判断下一步操作，而不是只看到模糊的失败结论。

### 6. 管理与运行观测界面持续改进

围绕运行详情页和管理界面，这一版本也包含了一批稳定性与一致性修复。

- 引擎升级后的版本号刷新更及时
- 聊天区的消息展示规则更加统一
- 折叠历史、标题、样式和不同视图之间的表现更加一致
- 运行详情页更适合用于回看执行过程和定位问题

### 7. 引擎支持矩阵调整

`iflow` 已在本版本中正式退役。

- 相关代码仍保留在仓库中作为封存参考
- 历史运行记录仍可查看
- 新任务、认证、升级和管理入口不再继续支持该引擎

如果当前仍在使用 `iflow`，建议尽快迁移到仍在维护的引擎。

## 兼容性与升级提示

- 如果你依赖交互式任务中的旧消息展示方式，升级后会看到更明确的“最终回复 / 折叠历史”语义
- 如果你使用需要用户多轮补充的技能，前端回复方式会更灵活
- 如果你仍在使用 `iflow`，需要提前规划迁移

本版本不引入新的对外 HTTP API 破坏性变更，但内部运行时和测试基础设施有较大调整，因此建议在升级后关注以下内容：

- 结构化输出任务的收敛行为
- 需要认证或等待用户补充信息的任务
- 管理界面中的运行详情展示

## 总结

`v0.6.0` 的核心价值，在于把过去几个版本快速演进过程中积累下来的关键运行时问题做了一次集中收敛。

与 `v0.5.2` 相比，这一版本更强调稳定性、一致性和可解释性。对于日常使用来说，最直接的体现是：结果更可靠，交互更自然，状态更清楚，问题也更容易定位。
