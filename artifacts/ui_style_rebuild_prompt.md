# UI 重建风格提示（仅风格，不含功能实现）

## 目标
- 先稳后美化：功能、路由、模板注入正确优先；视觉在不破坏行为前提下统一。
- 管理 UI 与 E2E UI 共享同一设计语言，但信息密度可按页面职责调整。

## 视觉基调
- 整体风格：专业、克制、开发者工具感，不走花哨营销风。
- 结构：深色导航/浅色内容区；主要信息放在浅色卡片中。
- 字体：`Inter, system-ui, -apple-system, "Segoe UI", sans-serif`。
- 圆角与阴影：中小圆角 + 轻阴影，hover 只做微弱抬升，不使用夸张动效。

## 设计 Token（建议）
- 主色（Primary）：蓝色系，按钮/链接/焦点态统一。
- 语义色：成功/警告/错误/信息独立，badge 与提示框语义一致。
- 中性色：正文、次级文字、分割线、面板背景要有清晰层级。
- 间距：8px 基础节奏（8/12/16/24/32）。
- 动效：150ms~250ms，hover/focus 过渡一致。

## 组件风格要求
- `card`：浅背景、1px 边框、轻阴影。
- `table`：表头与内容分层清晰，行 hover 高亮轻微。
- `btn`：主次危险三类统一样式与状态。
- `badge`：状态颜色明确，文本可读性优先。
- `form-control`：focus 态必须明显（边框+ring）。
- `sidebar/nav`：当前激活态明确，不依赖颜色唯一表达（可加背景/字重）。

## 布局要求
- Runs/Run Detail 等高密度页面：优先可读与可扫描。
- 长日志/协议区域：使用等宽字体与固定高度滚动区。
- 移动端不是主目标，但至少保证中等宽度不会断裂。

## 禁止项
- 不修改业务路由和 API 语义以迁就 UI。
- 不在模板里堆复杂多行 Jinja 表达式（尤其 JS 注入变量）。
- 不将 i18n key 与自然语言混写为同一体系。
- 不恢复 recordings 相关入口与链路。

## i18n 落地约束
- 模板统一使用 `t("namespace.key", default="...")`。
- JS 文案先注入常量，再在脚本使用，禁止脚本中直接拼复杂 Jinja。
- 缺 key 时允许显示 default，但不得裸露 `client.xxx`/`nav.xxx` 变量名。

## 重建优先级
1. 模板稳定（可编译、脚本注入正确、路由不漂移）
2. i18n 稳定（key/default 一致）
3. recordings 下线收口（无残留引用）
4. 视觉统一（设计系统 + 分页迁移）
